Faza 1: Gruntowny Audyt Kodu i Architektury (Faza Analizy)
Cel: Pełne zrozumienie istniejącego rozwiązania, jego mocnych stron i zidentyfikowanie wszystkich potencjalnych obszarów do ulepszenia. Jules ma działać jak doświadczony architekt oprogramowania.
Kluczowe Działania:
Analiza Strukturalna: Przeanalizuj całą strukturę projektu w branchu feniks. Zidentyfikuj główne moduły (parsery, chunkery, klient Qdrant, etc.) i oceń ich spójność oraz logikę powiązań (separation of concerns).
Audyt Jakości Kodu: Przeprowadź statyczną analizę kodu pod kątem:
Złożoności cyklomatycznej (wyszukanie zbyt skomplikowanych funkcji).
Potencjalnych "code smells" (duplikacja kodu, długie metody, nadużywanie globalnych zmiennych).
Spójności w nazewnictwie i stosowania konwencji TypeScript.
Dogłębna Analiza Logiki Ingestycji:
Parsowanie (AST/HTML): Sprawdź, czy parsery obsługują wszystkie edge-cases występujące w kodzie AngularJS (np. niestandardowe dyrektywy, złożone bindingi). Czy błędy parsowania są poprawnie obsługiwane?
Chunkowanie: To jest kluczowy punkt. Oceń, czy obecna strategia chunkowania jest optymalna. Czy nie gubi kontekstu? Czy metadane dołączane do chunków są wystarczająco bogate? Czy można ją ulepszyć, aby lepiej oddawała intencje kodu AngularJS?
Integracja z Qdrant: Zweryfikuj, czy komunikacja z bazą wektorową jest wydajna (czy używa batchowania?), bezpieczna (czy klucze API nie są w kodzie?) i odporna na błędy (czy jest mechanizm ponawiania prób?).
Wydajność i Zarządzanie Zasobami: Zidentyfikuj potencjalne wąskie gardła. Czy skrypty nie wczytują niepotrzebnie dużych plików do pamięci? Czy operacje można zrównoleglić?
Ocena Pokrycia Testami: Sprawdź, jakie jest obecne pokrycie kodu testami jednostkowymi i integracyjnymi. Zidentyfikuj krytyczne ścieżki logiki, które nie są testowane.
Dokumentacja: Oceń stan dokumentacji wewnątrz kodu (komentarze JSDoc) oraz istnienie pliku README.md wyjaśniającego, jak uruchomić i skonfigurować silnik.
Artefakt tej fazy: Jules przygotuje szczegółowy raport w formacie Markdown (AUDIT_REPORT.md) zawierający wszystkie powyższe ustalenia i rekomendacje.
Faza 2: Implementacja Ulepszeń (Faza Refaktoryzacji)
Cel: Systematyczne wprowadzanie poprawek zgodnie z raportem z audytu, w małych, logicznych i dobrze udokumentowanych krokach.
Kluczowe Działania:
Jules będzie implementował zmiany w oparciu o priorytety ustalone w raporcie.
Praca w Atomowych Commitach: Każda znacząca poprawka (np. refaktoryzacja modułu chunkującego, dodanie obsługi błędów, poprawa wydajności parsera) zostanie zamknięta w osobnym commicie z bardzo szczegółowym opisem, wyjaśniającym CO zostało zmienione i DLACZEGO ta zmiana jest ulepszeniem.
Rozbudowa Testów: Równolegle z refaktoryzacją kodu, Jules będzie rozbudowywał zestaw testów jednostkowych i integracyjnych, aby pokryć nowe funkcjonalności i zabezpieczyć istniejące przed regresją.
Czystość Kodu: Jules zadba o najwyższą jakość kodu, stosując zasady SOLID, DRY i dbając o czytelność oraz prostotę rozwiązania tam, gdzie to możliwe.
Dokumentacja na Bieżąco: Wszelkie nowe funkcje lub zmiany w API wewnętrznym będą na bieżąco dokumentowane w kodzie.

Cel i zakres
Główny cel: Poprawa projektu Fenioks którego celem jest Przygotowanie bazy wiedzy która umożliwi agentom AI w pełni-automatyczna migracja dużego kodu AngularJS (ok. dziesiątki tysięcy linii) do Next.js (z TypeScript, app router, SSR/SSG), przy jednoczesnym porządkowaniu architektury, zwiększeniu wydajności i testowalności.
Zródło danych dla projektu Feniks czylei jakie repo ma być poddane refaktoryzacji: https://github.com/dreamsoft-pro/frontend wyłącznie branch master
Zakres refaktoryzacji dla docelowego projektu: JS → TS, $scope/controller → komponenty/serwisy, ui-router/ngRoute → routing Next.js, ng-templates → React/JSX, dyrektywy → komponenty/dyrektywy React, filtry → utilsy/haki, serwisy $http → fetch/axios + SWR/React Query, budowa nowej warstwy UI (Tailwind/shadcn).
Wymagania biznesowe: zachować parytet funkcjonalny (pixel-parity tam gdzie ważne), uruchomić i18n, SEO, oraz stabilny pipeline CI/CD.
Architektura “Feniks”
repozytorium do poprawy : https://github.com/dreamsoft-pro/feniks wyłącznie branch main i ostatni commit
Ingest wiedzy (RAG/AST):
parsowanie repo (AngularJS, HTML, CSS, assety) → chunkowanie semantyczne (Node skrypty) + ekstrakcja AST (TS/ESTree) dla kodu,
selektywne “krojenie” na bloki: komponenty, serwisy, dyrektywy, trasy, widoki, filtry,
metadane domenowe: flow użytkownika, zależności między modułami, punkty wejścia (bootstrap AngularJS), mapy routingu.
Wektorowa baza wiedzy: Qdrant jako core (już masz w ekosystemie); kolekcje: code_chunks, route_maps, ui_snippets, domain_glossary.
Indeksy pomocnicze: graf zależności (kto kogo importuje, które widoki są osadzone), słownik i18n, taksonomia komponentów UI.
Agentura AI:
Agent Skaut – buduje plan migracji na podstawie mapy modułów.
Agent Refaktor – generuje patche (JS→TS, dyrektywy→komponenty React, filtry→utils).
Agent UI – przenosi layout do React + Tailwind (z zachowaniem klas/tematów).
Agent Testów – tworzy Playwright + Vitest/RTL, smoke i e2e.
Agent QA – porównania zrzutów ekranu (pixel-diff), Lighthouse/ Web Vitals, regresje.
Kontrola jakości: reguły ESLint/Prettier, TS strict, budżety wydajności (LCP/CLS/TTI), checklisty dostępności (WCAG), i testy porównawcze UI.
Pipeline migracji (zapamiętany plan krok po kroku)
Audyt i inwentaryzacja AngularJS
wykrycie modułów, dyrektyw, filtrów, tras, miejsc heavy-digest; graf zależności.
Stabilizacja AngularJS (przed migracją):
ograniczenie watchy, bind-once tam gdzie możliwe, separacja logiki od DOM; uproszczenie serwisów HTTP.
Warstwa danych i kontrakty:
opis interfejsów w TS (DTO) równolegle, nawet zanim padnie pierwszy komponent React; mapowanie endpointów.
Wycięcie wspólnych utili:
filtry/helpsy → funkcje czyste (TS) + testy jednostkowe.
Routing i “szkielet” Next.js:
nowy repo/module: Next.js (app router), TypeScript, Tailwind, shadcn/ui, layout, meta/SEO, i18n.
Migracja ekranami (strumienie funkcji):
listy → szczegóły → formularze, każdorazowo z testami i zrzutem porównawczym.
Warstwa danych w React:
fetch/axios + SWR/React Query, cachowanie, mutacje, standard obsługi błędów/toastów.
i18n:
centralny słownik, import istniejących języków; automatyzacja wykrycia stringów z widoków AngularJS.
Zgodność wizualna:
narzędzia do snapshotów (Playwright screenshot diff), budżety odchyleń.
Degradacja AngularJS:
wygaszanie moduł po module; stan “hybrydowy” możliwy (reverse proxy lub podścieżki).
Twarde QA i wydajność:
Lighthouse w CI, Web Vitals w prod, A/B jeśli potrzebne na krytycznych ścieżkach.
Cut-over:
przełączenie ruchu, fallback do starego widoku tylko dla edge-case (czasowe).

Używasz Docker + Kubernetes + Rancher (OVH), kontenery dev/prod, Jenkins buduje obrazy po utworzeniu brancha typu release_*.
Repozytoria w GitHub; często pracujesz na Windows/PowerShell oraz Linux/Docker, masz automatyzację deploymentu po branchu release.
Dążysz do pre-prod z testami e2e (Playwright) i regresją UI (pixel-diff) oraz Lighthouse w CI.
